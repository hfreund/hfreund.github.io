Application overview + condition

Built out of an evolution of a desktop application, this benefits administration webapp for human resource departments and insurance brokers was a challenge to navigate in even the best of conditions. The problems came from a few areas:

  1. Architecture: The system architecture and navigational structure were not designed to accomodate continuous additions of function and modules. Without formal training, new users were often unable to locate even basic tools.
  2. User Interface: Divergent efforts by a series of developers to create new layouts or update UI components resulted in a fractured user experience. Where module archetypes did exist, they tended to be many years out of step with contemporary practices.
  3. Dependencies: Many modules that integrated into the system could not be changed or updated easily due to dependencies or compliance requirements. Therefore, new development was frequently hamstrung to advance in a new direction, which would deviate from the dependent modules.

Working with the development team, I helped create a method for bringing the user interface up to date and designed the future system UI to work towards. The update would take place in two phases:

Phase I

Establish a common denominator for the UI, allowing for system-wide changes in the future.

Phase II

Progressively transition from the old UI to the new UI.

Establishing a common denominator

The first part of phase one was to identify and catalogue the myriad variations of UI elements. This gave me a good idea of how many element variations existed, which occured most frequently, and under what conditions they were used.

It turned out to be more complicated than merely identifying them in the front end, as some older ASP controls used entiely different implementations. It wasn't uncommon to find a button styled by the code behind instead of a class.

Phase I: Conversion to Bootstrap 3

The decision to use Bootstrap 3 as the baseline was driven by two factors. First, a large portion of the webapp had already been developed in Bootstrap 2. Second, the companion enrollment application was built in Bootstrap 3 and these two applications would ultimately share most of their modules, so having a common language made sense and would save development effort in the near and long terms.

The goal of this conversion was to create the least amount of noticeable impact on the user while giving our development team a solid platform off of which to build future modules. Therefore, any functional changes to the navigation flow or significant changes to master page layouts were off limits. I did keep in mind the future direction of the platform, using mockups for guideance during this first phase so there would be less of a gap to span with phase two.

Phase II: Architecture & layout

The second phase was a meaningful departure from the current architecture and layout typologies that were developed at a time when the standard browser windows varied between 600 x 800 and 1024 x 768. The update focused on organizing navigation and improving usability across browsers and devices.
