Transforming a Legacy Web App

Common Benefits was forward thinking in capability, however, in order to stay competitive with a bustling marketplace after the passage of the Affordable Care Act, the entire user experience and user interface needed to be refreshed.

The web app was built in an era of UI design dictated by fixed widths, tables, 4:3 aspect ratios, and early skeuomorphism, which meant that this project was essentially a clean slate with no former style guide to adhere to and the opportunity to set a tone for future development projects. There were still some legacy components that could not be changed in the initial launch due to shared dependencies and development constraints. I had to be mindful about including those components in their present condition in such a way that they would not appear out of place, while also planning for their future redevelopment down the road.

Not only was this project a serious front-end overhaul â€“ the development team planned a complete overhaul of the codebase including shifting from Web Forms to MVC Framework, using client-side JavaScript to handle more operations, and writing more extensible code to be reused through this and other applications being developed in parallel. At the beginning of the project my development chops were next to nil, so this presented a great opportunity to get my feet wet by working directly with senior developers.

Reduce and Refine

The primary goal of the first phase was to transition existing functionality to a contemporary application. The core user base was not a particularly tech savvy crowd, so great care was taken to maintain a familiar experience while ushering in a new interface with improved functionality. Members of the Client Services Team provided essential insight into our user groups and helped identify existing pain points.

Flexible, Focused Approach

Although we had control of the underlying framework, the software has to integrate with over one hundred benefit enrollment modules from different insurance carriers, each with their own compliance requirements and many which were custom made before a time of flexible modules. Here we were severely restricted since almost any change would require compliance approval which could dramatically set the project off track.

Universal Login

In the legacy app users were shunted between three login pages, each with its own style and a lot of distracting information. The new version adhered to a common design language, removed a lot of the bloat, and was fully responsive. Additionally, instead of diving into a sub-page to select the company, users could type a few letters and the system would provide their company to them.

We rolled out the new login UI several months ahead of the rest of the UI updates. Since there would be a necessary one-time account upgrade for users with existing accounts, we wanted to be sure they were comfortable with the login and password reset processes well in advance of when they might need to use them.

In a second phase, third party sign in was deployed for CBI users. This enabled expanded use by union workers, contract workers, affiliates, and other individuals who were not employees of a specific company.

Deployment

The beta deployment included select generic modules and was introduced to clients on a case-by-case basis. During this time, additional modules were converted and invluded for a followup deployment that would roll out to all clients over several subsequent months.
